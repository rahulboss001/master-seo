4.1

Site Speed & Core Web Vitals — Direct Ranking Factors

Intermediate ▼
Page speed has been a confirmed Google ranking factor for mobile since 2018 and for desktop since 2010. In 2021, Google made Core Web Vitals official ranking signals — the first time Google publicly announced a specific, measurable technical standard your site must meet to rank competitively. Speed improvements are unique in SEO: they deliver fast, measurable results AND simultaneously improve conversions. Amazon's internal data showed every 100ms of extra load time cost them 1% in sales. Speed is not just an SEO issue — it's a revenue issue.

LCP — Largest Contentful Paint (Target: <2.5s): Measures how long until the largest visible element fully loads — usually your hero image or main heading. The #1 cause of poor LCP is an unoptimised hero image. Fix: convert to WebP, compress below 100KB, add fetchpriority="high" and explicit width/height attributes, use a CDN.
INP — Interaction to Next Paint (Target: <200ms): Replaced FID in March 2024. Measures how quickly a page responds to user interactions (clicks, taps, typing). Poor INP causes: heavy JavaScript execution blocking the main thread, excessive third-party scripts (chat widgets, ad networks, analytics). Fix: defer non-critical JS, remove unused third-party scripts.
CLS — Cumulative Layout Shift (Target: <0.1): Measures visual stability — how much page elements move around during loading. The #1 frustration signal: a user is about to click a button, the layout shifts, and they click the wrong thing. Cause: images and ads without defined dimensions, web fonts causing text reflow. Fix: always specify width/height on images, reserve space for ads.
Field Data vs Lab Data: PageSpeed Insights shows both. Lab data (Lighthouse simulation) is useful for debugging specific issues. Field data (Chrome User Experience Report — real Chrome user measurements) is what Google actually uses for ranking signals. A site can score 30 in lab but have Good field data if its real users have fast connections.
Speed Budget by Element: Images typically account for 60–70% of page weight. The fastest speed wins come from image optimisation first, then render-blocking resources (CSS/JS), then server response time (TTFB). Follow this order of priority — don't optimise JavaScript before images.
Hosting & TTFB (Time to First Byte): Even perfect front-end optimisation won't overcome a slow server. Target TTFB under 800ms. Causes of slow TTFB: shared hosting overloaded, no CDN, database queries not cached, no page caching plugin (WordPress). Fix: upgrade hosting or add a caching layer (WP Rocket, Cloudflare).
📖 Real Impact Case Study

An e-commerce site had an LCP of 8.4s due to a 4.2MB uncompressed hero image. Fixes applied in order:

  1. Converted JPEG to WebP — saved 78% file size
  2. Added fetchpriority="high" + explicit dimensions
  3. Moved image to CDN (Cloudflare)

Result: LCP 8.4s → 1.6s. Mobile PageSpeed 28 → 91. Organic mobile traffic +34% in 3 months. Checkout conversion rate +12%. One image fix outperformed 6 months of content creation.

4.2

XML Sitemaps, Canonical Tags & Duplicate Content

Intermediate ▼
XML sitemaps, canonical tags, and duplicate content management form your site's indexing strategy — telling Google precisely which pages to crawl, which version is "official," and preventing your own pages from competing against each other. These three elements are deeply interconnected: a poorly managed sitemap allows duplicate URLs to be indexed; missing canonical tags let those duplicates split ranking power; and together they can quietly destroy months of SEO work. Getting this right is foundational for any site above 50 pages.
XML Sitemap — Your crawl priority blueprint: A structured file at yoursite.com/sitemap.xml listing every URL you want Google to crawl and index. Include: canonical pages, last-modified dates. Exclude: 404 pages, noindexed pages, redirect URLs, duplicate content pages, thin content (<300 words), admin/login pages. Submit in GSC → Sitemaps for priority crawling. Large sites should use a sitemap index (a sitemap of sitemaps).
Canonical Tag — Solving the "which version?" problem: <link rel="canonical" href="https://example.com/preferred-url/"> in the <head> tells Google "this is the authoritative version — consolidate all signals here." Critical for e-commerce sites where product filters create hundreds of URL variants (/shoes?color=red&size=10&brand=nike) that are technically "different" URLs but show the same products.
Duplicate Content — The silent ranking killer: When the same content is accessible at multiple URLs, Google must decide which to rank. It usually picks neither well. Common sources: HTTP vs HTTPS, www vs non-www, trailing slash vs none, URL parameters from analytics or session tracking, pagination, printer-friendly versions, syndicated content. None of these require the same content to rank well independently.
Pagination — rel="next" and rel="prev" (deprecated) → canonical to "View All": Google deprecated pagination rel attributes in 2019. Modern approach: either use a "View All" canonical page or ensure pagination pages have unique enough content (different products shown) to stand alone. Never block paginated pages in robots.txt — they carry link equity from product pages.
URL Parameter Handling: Use Google Search Console → Legacy tools → URL Parameters (or simply canonical tags) to tell Google which parameters change content (?sort=price) vs which are tracking only (?utm_source=newsletter). Tracking parameters should be canonicalised to the clean URL to avoid splitting signals.
📖 Real Case Study — Canonicals

An e-commerce site's /blue-running-shoes page was accessible at 4 versions: HTTP, HTTPS, www, non-www. Plus faceted navigation created 500+ filter URL variants. Google was splitting authority across 500+ "different" pages — the main page received only 0.2% of its potential link equity.

After implementing canonical tags pointing all variants to the primary HTTPS/www URL and blocking parameter URLs via robots.txt, main product pages jumped an average of 8 positions within 60 days. Zero new content or backlinks needed.

4.3

Redirects, HTTPS, Mobile-First Indexing & Site Architecture

Advanced ▼
These four technical elements form the architectural skeleton of a healthy, well-ranking website. Unlike content, which produces gradual compounding gains, errors in any of these four areas can silently drain ranking potential for months without clear symptoms — making them some of the most dangerous and underdiagnosed technical SEO issues. Mastering them separates junior SEOs from technical professionals.
301 vs 302 Redirects — Choose correctly every time: 301 = Permanent redirect. Passes ~99% of link equity to the destination. Use when: permanently moving a URL, consolidating duplicate URLs, domain migrations, deleting pages (redirect to the most relevant alternative). 302 = Temporary redirect. Google typically does NOT transfer full link equity. Use only when a redirect is genuinely temporary (e.g., A/B testing, seasonal content temporarily moved).
Redirect Chains — The hidden link equity drain: When A → B → C → D, each hop loses a fraction of link equity. After 3+ hops, the loss becomes measurable. Any site that has migrated platforms, rebranded, or changed URL structures multiple times likely has chains. Audit with Screaming Frog → Reports → Redirects → Chains. Fix to direct A → D.
HTTPS — Non-negotiable since 2014: Every new site must launch with HTTPS. Google shows a "Not Secure" warning in Chrome for HTTP sites, destroying user trust. HTTP→HTTPS migration checklist: (1) Install SSL certificate, (2) 301-redirect every HTTP URL to HTTPS, (3) update all internal links, (4) update sitemap to HTTPS URLs, (5) add new HTTPS property in Google Search Console, (6) update Google Analytics property.
Mobile-First Indexing — Since 2023, all sites are mobile-first: Google uses your site's mobile version as the primary version for indexing AND ranking — including for desktop searches. If your mobile version has less content than desktop (hidden content, truncated text, missing images), your rankings suffer on all devices. Test with Google's Mobile-Friendly Test. Ensure responsive design (not separate m.site.com).
Site Architecture & Crawl Depth: Every page should be reachable within 3 clicks from the homepage. Pages at click depth 4+ receive proportionally less crawl budget and link equity. Flat architecture: Homepage → Category → Page (depth 2) is ideal. Deep architecture: Homepage → Category → Subcategory → Sub-subcategory → Page (depth 4+) starves deep pages of crawl and authority.
⚠️ Redirect Chain Disaster — Real Case

A site migrated domains 3 times over 5 years: domain-A → domain-B → domain-C → domain-D. Result: a 4-hop chain for every backlink pointing to the original domain.

Their 4,000 backlinks were passing only ~70% of potential link equity by the final destination. After consolidating all chains to direct 301s (A→D, B→D, C→D), rankings recovered measurably within 60 days with zero new content or backlinks.

Lesson: Any site that has changed its URL structure or domain in the past 5 years should run an immediate redirect chain audit. The equity loss accumulates silently.

4.4

Crawl Budget, robots.txt & Log File Analysis

Advanced ▼
Crawl budget is the number of pages Google's crawler (Googlebot) will crawl and index on your site within a given timeframe. For small sites (<500 pages), crawl budget is rarely a concern. For large sites (10,000+ pages), it's critical — Googlebot may never reach your most important content if it wastes crawl quota on low-value pages. robots.txt is your primary lever to direct that budget. Log file analysis is how you see exactly what Googlebot is doing on your site — the most powerful and most underused technical SEO technique.
Crawl Budget — What determines it: Crawl Rate (how fast Googlebot can crawl without overloading your server) × Crawl Demand (how often Google wants to recrawl based on popularity and freshness). Googlebot crawls high-authority, fast, well-linked sites more frequently. A slow server literally reduces how many pages Google will crawl per day.
robots.txt — Directing the crawl: Located at yoursite.com/robots.txt. Tells crawlers which paths are allowed or disallowed. IMPORTANT: Blocking a URL in robots.txt does NOT remove it from the index — it just stops Googlebot from recrawling it. To block pages from the index, use a noindex meta tag (but allow Googlebot to crawl the page to see the noindex).
What to Block in robots.txt: Admin areas (/wp-admin/), staging environments, search result pages (/search?q=), infinite pagination (?page=500+), duplicate URL parameter variants, checkout/cart pages, internal search. This frees budget for pages that should rank.
Log File Analysis — The ground truth: Your web server logs every single request — including every Googlebot visit. Log analysis reveals: which pages Google crawls most often (crawl frequency), which pages are never crawled (orphans or architecture issues), which return errors during crawling (404, 500s), and how Googlebot's visit frequency compares to your publishing frequency. Tool: Screaming Frog Log File Analyser.
💻 robots.txt Example
User-agent: *
Disallow: /wp-admin/
Disallow: /search/
Disallow: /cart/
Disallow: /checkout/
Disallow: /*?sort=
Disallow: /*?filter=
Disallow: /tag/
Allow: /wp-admin/admin-ajax.php

Sitemap: https://example.com/sitemap.xml

⚠️ Never disallow your CSS and JS files — Googlebot needs them to render pages correctly. Always include your sitemap URL at the bottom.

4.5

JavaScript SEO & Structured Data Deep Dive

Advanced ▼
JavaScript SEO is one of the most misunderstood and dangerous technical SEO areas. Modern websites rely heavily on JavaScript for rendering content — but Googlebot processes JavaScript differently from how a browser does, and critically, it can introduce significant indexing delays of days or weeks. Structured data (schema markup) connects to JS SEO because it's often injected via JavaScript — meaning if Googlebot can't render the JS, it can't see the schema either.
How Googlebot Handles JavaScript (2024): Googlebot renders JavaScript, but in a two-stage process: (1) First crawl: HTML only, without rendering JS. (2) Second pass: JS rendering queue — this can take days to weeks. Content that only exists after JS execution (SPA apps, infinite scroll, JS-injected text) may be invisible or significantly delayed in indexing.
Server-Side Rendering (SSR) vs Client-Side Rendering (CSR): SSR sends complete HTML to Google (fast indexing, no delays). CSR sends a bare HTML shell then renders content via JS (indexing delays, potential content invisibility). For SEO-critical pages (product pages, blog posts, category pages), SSR or Static Site Generation (SSG) is strongly preferred.
Diagnosing JavaScript Issues: Use "Inspect URL" in Google Search Console to see what Googlebot sees. The screenshot shows the rendered page — if your content appears there, it's indexed. If the screenshot shows a blank or incomplete page, you have a JS rendering problem. Also check: "View Source" vs browser DevTools Elements — if content is in Elements but not Source, it's JS-rendered.
Structured Data (Schema) Best Practices: Always implement schema as JSON-LD (not Microdata or RDFa) — Google officially recommends it, it's cleaner, and it's independent of HTML structure. Validate every schema implementation in the Rich Results Test before deploying. Common eligible types for rich results: FAQ, HowTo, Recipe, Product, Review, Event, JobPosting, Article, BreadcrumbList.
Dynamic vs Static Schema: If you use a headless CMS or React/Vue/Next.js, inject JSON-LD schema in the <head> server-side (or in Next.js's next/head component) to ensure it's present in the initial HTML response — never rely on client-side JS injection for schema markup on SEO-critical pages.
💡 JS SEO Diagnostic Steps
  1. Open page in Chrome. Press Ctrl+U (View Source). Search for your main content. Is it in the raw HTML?
  2. If NO → content is JS-rendered. Check if Google sees it via GSC Inspect URL → "Test Live URL" → screenshot.
  3. In GSC, check Coverage report → "Crawled but not indexed." JS rendering issues often show as this status.
  4. Use Fetch as Google (GSC → URL Inspection → Request Indexing) to force a re-render and see if the issue resolves.
  5. If persistent: implement SSR or pre-rendering for the affected pages. Consider using a service like Prerender.io for SPAs.

4.6

Full Technical SEO Audit — Professional Workflow

Advanced ▼
A technical SEO audit is a comprehensive health-check of a website's technical infrastructure — and it is the single most valuable deliverable you can offer as a freelance SEO professional. A well-executed technical audit: identifies every issue blocking rankings, provides clear prioritised action items, demonstrates measurable before/after improvement, and serves as proof of skill when pitching clients. This topic gives you the complete professional workflow to conduct one.
Phase 1 — Crawl & Data Collection: Run Screaming Frog on the full site (or sample 1,000 URLs for large sites). Collect: all status codes, titles, meta, H1s, word counts, canonical tags, hreflang, internal links, image alt text, page speed (via PSI API integration), redirect chains. Simultaneously run Ahrefs Site Audit and GSC Coverage report for additional signal layers.
Phase 2 — Categorise Issues by Severity: Critical (blocking indexing): 5xx server errors on important pages, incorrectly blocked robots.txt, noindex on key pages, missing canonical tags on paginated content. High (significantly impacting rankings): redirect chains 3+ hops, slow CWV scores (>4s LCP), duplicate H1/title tags, thin content on indexable pages. Medium (incremental impact): missing meta descriptions, images without alt text, missing schema on eligible pages.
Phase 3 — Prioritise by Traffic Impact: Sort all affected pages by their organic traffic in GSC. Fix issues on high-traffic pages first — the same fix on a 10,000-visit/month page delivers 100× the value of fixing it on a 100-visit/month page. This is what clients pay for: ROI-focused prioritisation, not just a list of issues.
Phase 4 — Implement & Measure: Record the date of every fix in a change log. Wait 4–8 weeks (Google's re-indexing cycle). Compare before/after in GSC: impressions, clicks, average position for affected pages. This is your case study — screenshot everything before you start.
📖 Audit Deliverable Structure

A professional technical audit report contains:

  1. Executive Summary — 1 page, key findings, overall health score
  2. Critical Issues — Each with: what, why it matters, how to fix, priority score
  3. High Priority Issues — Same format
  4. Medium/Low Issues — Summary format
  5. Implementation Roadmap — Week 1, Week 2–4, Month 2–3 plan
  6. Before/After Baseline — GSC screenshots, crawl data export

This format justifies a ₹15,000–₹50,000 one-time audit fee. The roadmap shows ongoing retainer potential.

🎯 Module 4 Quiz — Technical SEO

1. What is the target LCP score for a "Good" Core Web Vitals rating?

AUnder 1.0 second
BUnder 2.5 seconds
CUnder 4.0 seconds
DUnder 6.0 seconds
✅ Correct! LCP under 2.5s = Good. 2.5–4s = Needs Improvement. Over 4s = Poor (ranking penalty risk on mobile). LCP is the most impactful CWV metric to fix first.
❌ The correct answer is under 2.5 seconds. This is Google's "Good" threshold for LCP. Anything over 4s triggers the "Poor" classification and can result in ranking penalties on mobile.

2. A page is blocked in robots.txt. What happens to it in Google's search results?

AIt is immediately removed from the index
BIt may still be indexed if other sites link to it — but Google can't see its content
CIt is permanently de-indexed
DNothing — robots.txt has no effect on Google
✅ Correct! Robots.txt blocks crawling, NOT indexing. A blocked page can still appear in search results if other sites link to it — Google will show the URL without a snippet ("No information is available for this page.") To prevent indexing, use a noindex meta tag instead.
❌ Robots.txt blocks crawling, not indexing. A blocked page can still be indexed via external links — Google just can't read its content. To truly de-index a page, use a noindex meta tag (and allow Googlebot to crawl it to see the tag).

3. You need to permanently move a URL. Which redirect type should you use?

A301 — Permanent redirect (passes ~99% link equity)
B302 — Temporary redirect
C307 — Temporary redirect (HTTP/1.1)
DMeta refresh redirect
✅ Correct! A 301 redirect signals a permanent move to Google and passes approximately 99% of the original page's link equity to the destination. Using a 302 for a permanent move risks Google not fully transferring the authority.
❌ For permanent URL moves, always use a 301 redirect. It signals permanence to Google and transfers ~99% of link equity. A 302 is for temporary redirects only and may not fully pass authority.

4. What does a canonical tag do?

APermanently redirects the page to another URL
BTells Google which version of a page is the "official" one to consolidate signals toward
CBlocks the page from being crawled
DRemoves the page from the index
✅ Correct! A canonical tag (<link rel="canonical">) tells Google "this is the preferred URL — consolidate all signals (backlinks, ranking data) to this version." It doesn't redirect users, it doesn't block crawling — it instructs Google which version to rank when duplicates exist.
❌ A canonical tag tells Google which URL is the "authoritative" version when duplicates exist. It consolidates all ranking signals to the canonical URL without redirecting users. It's different from a redirect (user movement) or noindex (removal from search).

5. Google now uses which version of your site for indexing and ranking?

AThe desktop version
BThe mobile version (mobile-first indexing)
CWhichever version loads fastest
DBoth versions equally
✅ Correct! Since 2023, all sites use mobile-first indexing by default. Google crawls and indexes the mobile version of your site. This means if your mobile version has less content than desktop, your rankings on ALL devices suffer.
❌ Google uses mobile-first indexing for all sites. The mobile version is used for both indexing and ranking — even for desktop searches. If your mobile site has stripped content vs your desktop, all your rankings suffer.
